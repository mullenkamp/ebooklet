# Critical design review — pipelined push + progress logging (ebooklet 0.10.1, booklet 0.12.8)

**Reviewer:** independent Claude subagent, fresh context · **Date:** 2026-07-14
**Basis:** full source read of `~/git/booklet` (main.py, utils.py) and `~/git/ebooklet` (main.py, utils.py, journal.py, errors.py, fsck.py, remote.py, tests), portalocker 2.x installed source, downstream call-site grep of `cfdb/edataset.py` and `envlib/catalogue.py`, plus three executed experiments against the actual source trees (details in §4). All line numbers cited in the brief were re-checked against the current source and are accurate.

## 1. Verdict

**Implementable with changes.** The core mechanism is sound — the append-only offset invariant is real (verified empirically as well as by source), sync-first capture is sufficient, the fused-worker + gate topology is deadlock-free, and the semantics are reproducible on `fake_s3`. But three MAJOR items must be resolved before implementation: (1) the phase-A/loc_map merge precedence is unspecified and the wrong choice silently regresses remote data; (2) the pulled-bytes RAM ceiling is unbounded in exactly the stale-local scenario this feature targets (answer to C9: yes, this needs a redesign — recommended fix below dissolves finding 1 as well); (3) the compaction guard as specified does not cover the capture window, leaving a hole in which phase B commits wrong bytes undetected.

## 2. Findings

### MAJOR

**F1 — Phase-A merge precedence is unspecified; loc_map-first packing silently regresses remote data.**
Evidence: `ebooklet/utils.py:932-939` — a key that is locally present but older than the remote **is** pulled (the `:934` warning exists precisely for this state), so a key can simultaneously have a loc_map entry (stale value, stale ts) and a phase-A pulled entry (newer value). Experiment 1 proved the captured offset still yields the **old** bytes after an overwrite (append-only means the pull's materialization never touches the old block); Experiment 2 proved today's push packs the **pulled** value in this state (repacked group carried `a-NEW`, not the stale `a-old`).
Failure scenario: client B holds a stale copy of key K (ts 100); the remote has K at ts 200. B edits a sibling key in K's group and pushes. Phase A pulls K=V₂₀₀. If the phase-B worker resolves members loc_map-first, it packs V₁₀₀ and stages ts 100 → the commit replaces the remote's newer value with the stale one. Every length/verification check passes; nothing detects it. On the public commons this is silent published-data regression.
Resolution: state explicitly that **pulled bytes take precedence over loc_map for the same key** (and that the staged ts for such keys is the pulled ts). Better: adopt the F2 resolution, which removes the dual-source merge entirely.

**F2 — (Answers C9) Pulled-bytes RAM is unbounded; worst case ≈ the whole remote.**
Evidence: `ebooklet/utils.py:951-977` (phase-A executor) fully completes before the phase-B executor at `:1003` starts — the two `with ThreadPoolExecutor` blocks are strictly sequential. Under the design, every pulled value for every affected group is held in memory from pull completion until its group packs, i.e. all of them coexist before phase B drains any.
Failure scenario: a stale local file (or a fresh `flag='w'` file created from init_bytes, which materializes nothing) pushes one edited key per group on the 20 GB / 149-group production dataset → phase A pulls ~20 GB into RAM → OOM on the exact workload that motivated the redesign. Today's code has no such ceiling because pulls materialize to disk and packs re-read.
Resolution (recommended): drop the in-memory side channel. After phase A completes, if any pull happened, run **one `locations()` re-sweep** (sync-first, so the phase-A `set()`s are flushed and indexed) to rebuild loc_map; then phase B reads *everything* through the private fd uniformly. This (a) restores today's RAM profile (≤ threads payloads), (b) dissolves F1 — the live chain resolves each key to exactly one block, and the pulled (newer) value wins by construction because the overwrite killed the old block, (c) reads pulled values back through the page cache (they were just written — near-free), and (d) costs one extra header-only sequential sweep only on pushes that pulled (fresh pushes: zero). The alternative (byte-cap + per-key fallback reads for overflow) works but keeps the dual-source merge and its precedence rule alive. The empty-`b''` members need no bytes either way — the index length field is the value.

**F3 — The compaction guard does not cover the capture window; the belt snapshot must be taken at capture time, and `_push_active` must span `build_changelog`.**
Evidence: design §3c sets `_push_active` around `update_remote` only, but the capture happens earlier, inside `create_changelog` (called from `Change.push` at `ebooklet/main.py:265`, or independently via the public `Change.build_changelog`, `:128-137`). Design §2b leaves *when* the `compaction_count` snapshot is taken unspecified.
Failure scenario: a concurrent thread calls `prune()` between the changelog sweep and `update_remote` entry (or between sweep and a snapshot taken at update_remote entry). Experiment 1 shows prune relocates essentially every block (0/69 captured offsets still valid) while reads at old offsets return exactly `value_len` bytes of *other keys' data*. The flag isn't set yet, so `prune()` doesn't raise; the snapshot taken post-prune matches at pack time; phase B packs garbage; the push **commits** it. Silent remote corruption.
Resolution: (a) snapshot `local_file.compaction_count` **inside `create_changelog`, before/under the same lock acquisition that starts the sweep** — booklet's `_mutation_count` iterator guard (prune bumps it, `booklet/main.py:632`) already guarantees a mid-sweep prune raises, so a sweep that completes proves the snapshot spans it; return the snapshot alongside `loc_map`. (b) Set `_push_active` around the whole push body (before `build_changelog` at `main.py:265`, through phase D), not just `update_remote`. With both, the guard has no window.

### MINOR

**F4 — Gate scope: a module-global `_pack_gate` contradicts the per-open `push_packers` kwarg.**
The sketch (`with _pack_gate: # BoundedSemaphore(push_packers)`) reads as module-level state. envlib routinely holds several open datasets in one process; two concurrent pushes would share whichever gate was constructed first (one handle's `push_packers=10` silently overrides another's `1`), and pushes against *different* disks would serialize against each other. Resolution: construct the semaphore per `update_remote` invocation (or per-ebooklet instance at open), passed into the workers.

**F5 — (Answers C6) The abort-on-compaction-mismatch routing is the right call, but three details need specifying.**
Raising pre-commit is correct: a detected compaction is an invariant violation (the 3c flag should make it unreachable), not an operational failure, and per-group "retry" semantics would be false comfort since every unpacked group is equally suspect. Nothing is staged or committed; already-PUT generations are standard invisible orphans (fsck sweeps them — confirmed against `fsck.py`). Specify: (a) the coordinator must stop submitting / drain outstanding workers before raising (otherwise they keep PUTting orphans into a doomed push); (b) use a typed exception (`Error` subclass — a sibling of `PushInProgressError`, e.g. `ConcurrentCompactionError`), since callers distinguish raise-channels from `PushResult` failures (documented contract in `PushResult`, `main.py:70-76`); (c) the raise must precede the staged-index build (`utils.py:1067`), not merely phase C.

**F6 — Timestamp-0 truthiness edge in the loc_map comparisons (C5/C11 edge).**
`check_local_vs_remote` (`utils.py:661-671`) treats a local ts of `0` as *absent* (`if local_time_int:`), the lost-keys test at `:987` treats it as *present* (`is None`), and the staged-entry loop at `:1022` (`if ts:`) skips it. A presence-based `key in loc_map` test changes behavior for keys explicitly written with `timestamp=0`. Vanishingly rare, but the replacement comparisons should reproduce today's exact truthiness per site (use `loc_map.get(key)` and mirror each existing check) or the divergence should be documented.

**F7 — `threads=1` gives zero pack/upload overlap in the fused topology (unlisted).**
With one worker, pack and PUT strictly alternate — the gate rationale ("while one worker reads, up to nine others are PUTting") assumes threads ≥ 2. `S3Connection` defaults to `threads=10` (`remote.py:674`) so the default is fine, but a user-configured `threads=1` silently loses all pipelining. Worth one docstring sentence; not worth code.

**F8 — Windows mandatory-lock denial is range-limited, not whole-file (refines C8).**
portalocker's `Win32Locker` locks bytes `[0, 0xFFFF0000)` (~4.29 GB) via `LockFileEx` (`portalocker/portalocker.py:105,143`, `_lock_bytes_low = -0x10000`); the msvcrt fallback locks only the first 64 KB (`:188`). Consequences: (a) detection is per-*read*, not per-open — a read below the boundary raises `PermissionError` (⊂ OSError, winerror 33), one above it **succeeds** and returns correct bytes; (b) on a > 4.29 GB local file, some groups fast-path fine while others fall back — the design's per-group OSError fallback handles this correctly, but the "one warning per push" message should not imply the whole push fell back; (c) with the msvcrt locker, the fast path may work essentially everywhere (values live above 64 KB). No correctness issue — the design's fallback shape is right; adjust the prose and the warning text.

### NOTE

**N1 — C3's mutation inventory is incomplete (both additions harmless).** Inside `update_remote`, phase C may also call `local_file.set_metadata(...)` (skew-normalization in `_build_meta_section_for_push`, `utils.py:832`) — an append + index rewire, after phase B completes; and `local_file._set_file_timestamp(...)` (`:1112`) writes 7 header bytes at position 42. Neither moves value bytes; list them so the C3 argument is airtight.

**N2 — loc_map itself is O(n_keys) RAM.** ~128k keys ≈ tens of MB (fine); a multi-million-key file ≈ GBs if stored as `{str: (int,int,int)}`. Cheap mitigation if it ever matters: pack the value as 15 bytes (the index-entry encoding). Not a blocker for the current datasets.

**N3 — `locations()` ts contract diverges from `timestamps()`.** `timestamps()` raises `ValueError` on `init_timestamps=False` files (`booklet/main.py:377-378`); the design yields `None` instead. Defensible (offset iteration is still meaningful without ts), but document the deliberate divergence; ebooklet never hits it.

**N4 — BytesIO-backed booklets have no path for the private fd.** ebooklet always uses real paths, but `Booklet` supports BytesIO (`_is_file=False`, `utils.py:1224-1237`). The ebooklet worker should route through the fallback when `not local_file._is_file`, and `locations()` itself works fine on BytesIO.

**N5 — Logger move.** The phase-A pull line currently emits on `ebooklet.utils` (`utils.py:21`); moving it to `ebooklet.push` changes the emitting logger name. Parent-logger (`ebooklet`) configurations still capture it; cfdb/envlib reference no logger names (grep-verified). Changelog mention only.

**N6 — Test-design refinements for C12.** (a) "Content-identical push" must compare *unpacked entry sets + per-key index resolution*, not raw group-object bytes — entry order changes by design, and generation tokens are random. (b) Factor the worker's private-fd open into a patchable helper (e.g. `_open_private_reader(path)`) so the Windows fallback is exercisable on Linux by monkeypatched OSError. (c) The guard test can set `_push_active` directly, plus one threaded prune-during-push test for the belt.

**N7 — Positive: the "timestamp tightening" fixes a real (self-healing but ugly) race, not cosmetics.** Today the staged ts is re-read *after* upload (`utils.py:1021`); a user-thread overwrite between pack and bookkeeping stages a ts newer than the packed bytes — the committed index then claims T_new for V_old bytes (the journal's skew branch eventually repairs it on the next push, but the interim remote state is internally inconsistent). Packed-ts ≡ staged-ts by construction eliminates this class. Similarly, under mid-push user mutation the new design is snapshot-consistent where today's is read-at-pack-time; C11's "content-identical" holds exactly for quiescent pushes, and differs only in the safer direction under concurrency.

**N8 — Benchmark scope.** The changelog sweep's IO pattern also changes: today's `timestamps()` sweep reads *every value byte* (`iter_keys_value_from_start_end_pos`, `utils.py:565`, reads ts+key+value even with `include_value=False` — i.e. today's changelog pass streams the whole 20 GB); header-only converts that to ~n_keys strided small reads. Almost certainly a wash-to-win on HDD (ascending order, readahead), but include the changelog phase in the pre-release benchmark alongside the pack-gate widths.

**N9 — §4 alternatives: the calls are right.** The fused+gate vs queue analysis holds: per-group sorted sweeps stride the *whole* file (group membership is hash-scattered), so the push does ~149 elevator sweeps rather than one global sweep — a queue topology fed in global offset order would do a single sweep, but each per-group sweep (~15-25 MB/s by seek arithmetic, consistent with the design's estimate) already exceeds any plausible uplink, so the extra complexity buys nothing. The standalone bulk-lookup rejection is also right (chain walk = 2-3 random reads/key, verified in `get_last_data_block_pos`/`get_value_ts`).

## 3. Confirmations (C1-C12)

- **C1 — CONFIRMED** (source + experiment). Swept every `file.write` site in booklet: `update_index` zeroes only the old block's next_ptr and rewires chain pointers (`utils.py:995-1013`); `assign_delete_flag` likewise (`:885-888`); `reindex` appends the new bucket index and rewrites only next_ptrs, with skip blocks confined to the *old index region* which never contains data blocks (`:246-252`); `set_timestamp` writes 7 bytes at a fixed intra-block position (`:460-464`); `write_data_blocks`/`flush_data_buffer` append at EOF; `set_reserved`/`set_metadata` route through the same appenders (`main.py:97-100, 165-168`); header writes (`n_keys`, file timestamp) touch bytes < 200. Only `prune_file` (`:1052-1194`) and `clear` (`:1029-1049`) move/destroy value bytes; both bump `_compaction_count` (`main.py:633, 716`). Experiment 1: captured offsets byte-stable across overwrite/delete/auto-reindex/reserved-slot rewrite/metadata/set_timestamp; 0/69 survive prune; clear bumps the counter.
- **C2 — CONFIRMED** (source + experiment). Write-mode opens are `buffering=0` (`utils.py:1279, 1378`); sync flushes buffer + index under the lock (`main.py:786-797`). Experiment 1 read every captured offset through an independent plain `open(path,'rb')` with no fsync anywhere — all correct. No window exists in which a *returned* offset references unwritten bytes: `locations()` runs post-sync, and blocks buffered after capture occupy new offsets never captured.
- **C3 — CONFIRMED with two additions** (source). Replace-pending purge confirmed at `main.py:252-263`, strictly before `build_changelog` (`:265`). Full inventory of local-file mutations inside push: skew `set_timestamp` (`utils.py:736`, in-map), phase-A `set()` materializations incl. the `b''` branch (`:638, :949`), journal/remote-state `set_reserved` persists, **plus** the two the brief omitted (N1): phase-C `set_metadata` skew-normalization (`:832`) and the `_set_file_timestamp` header write (`:1112`) — all append-only or header-only. Concurrent *user-thread* mutations are outside C3's frame; sets are append-safe (snapshot semantics), prune/clear are the guard's job → F3.
- **C4 — CONFIRMED** (experiment). A prototype implementing the spec's exact scan (header read, value seek-past, reserved-key exclusion, two-region relocated layout) yielded exactly the live user keys — no dead/superseded blocks, no metadata/reserved keys — after overwrites, deletes, a real auto-reindex (13→12007 buckets), and after prune's relocated layout; offsets read back the correct current values via an independent fd; buffered-at-scan-time entries covered by sync-first.
- **C5 — CONFIRMED with one edge** (source). loc_map keys ≡ `local_file.keys()` at `:901` (same live-key enumeration, metadata/reserved excluded, nothing mutates the key set between capture and `update_remote` within push); `b''` members enter the pulled structure → not lost, matching today's post-materialization `get_timestamp is not None`; locally-deleted-but-remotely-present keys behave identically (pull → resurrect → pack) in both. Edge: ts=0 truthiness (F6).
- **C6 — CONFIRMED, routing endorsed with conditions** (F5). Every listed failure maps onto the existing `(error, offsets)` channel; the compaction mismatch correctly aborts all-or-nothing pre-commit.
- **C7 — CONFIRMED** (source + reasoning). Each task acquires the gate exactly once via `with` (released on `GroupTooLargeError` etc.); no lock nesting, no dependency cycle with the executor → no deadlock; PUTs run outside the gate → no starvation; `push_packers ≥ threads` means the semaphore can never block (≤ threads concurrent holders needed) → today's behavior. Caveats: F4 (gate scope), F7 (threads=1).
- **C8 — CONFIRMED with refinement** (source + experiment). POSIX: portalocker's default is `fcntl.flock` (`portalocker.py:435`), OFD-scoped — experiment confirmed a private fd open/close does **not** drop the exclusive lock (a second process is still denied `LOCK_EX|LOCK_NB`). Windows: `LockFileEx` denial surfaces as `PermissionError` ⊂ OSError on the *read*, per-handle even intra-process — but only within the locked range (F8). "Cannot silently read wrong bytes": holds **only after F1 and F3 are resolved** — a stale-but-valid offset is precisely a right-length wrong-bytes read that no length check catches.
- **C9 — ANSWERED: yes, a cap/redesign is needed** (F2). Recommended: post-phase-A `locations()` re-sweep; falling back to booklet reads for pulled keys is the acceptable second choice (page-cache-hot, as the brief surmises).
- **C10 — CONFIRMED** (source + reasoning). The bookkeeping loop is single-threaded (`as_completed`, `:1015-1028`); emitting per-group records there gives exact counts/totals with no shared-counter race; Python logging is internally locked; pulled-member sizes are available exactly from the index length fields before phase A, so the start-record totals can be exact.
- **C11 — CONFIRMED for quiescent pushes** (source + experiment). Readers resolve strictly via per-key offsets returned at pack time, verified against embedded entry headers (`get_remote_group_values`, `:617-635`); `recover_group_members` parses the whole self-describing object (`:565`); fsck and copy_remote are manifest-driven — nothing depends on intra-object entry order. Experiment 2 established the baseline end-state (index ts ≡ packed ts, offsets resolve, stale-local member repacked with the pulled value). Under concurrent mid-push mutation the new design deviates — in the safer direction (N7).
- **C12 — CONFIRMED** (source). `fake_s3` provides everything required (ranged 206/416 GETs, atomic PUT, listing, metadata round-trip, monkeypatch-friendly sessions — existing tests already inject PUT failures); all listed tests are hermetically achievable with the N6 refinements (comparison must be entry-set-based; factor the private-fd open for OSError injection).

## 4. Experiments run

All under the session scratchpad in /tmp, executed with `~/git/ebooklet/.venv/bin/python` with `sys.path` pointed at the **local source trees** (verified: `booklet.__file__` = `~/git/booklet/...`, version 0.12.7; `ebooklet.__file__` = `~/git/ebooklet/...`):

1. **`exp1_offsets.py`** — offset stability + `locations()` prototype fidelity on real booklet. 16/16 checks passed: capture = exact live user-key set; independent-fd reads correct post-sync without fsync; offsets byte-stable across overwrite/delete/auto-reindex (real 13→12007 reindex)/reserved-slot rewrite/metadata/set_timestamp; **overwritten key's captured offset returns the OLD bytes** (the F1 hazard, demonstrated); prune invalidates 69/69 offsets and bumps `_compaction_count`; post-prune two-region rescan correct; clear bumps the counter.
2. **`exp2_push_semantics.py`** — end-to-end grouped push on `fake_s3` with real ebooklet+booklet. 9/9 checks passed: with a stale local value and newer remote, today's push pulls and repacks the **newer** value (the baseline F1's merge must preserve); untouched members survive repack; committed index ts ≡ packed entry ts and offsets resolve into the group object; fresh reader sees the correct end state.
3. **flock check** (inline) — portalocker POSIX locker is `fcntl.flock`; opening and closing a private read fd on a portalocker-EX-locked booklet file does not release the lock (second-process `LOCK_EX|LOCK_NB` still denied).
