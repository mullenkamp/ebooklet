# Critical design review — ebooklet 0.9.5 (Phase 0 hardening), pre-implementation

**Scope note:** Plan mode was active, so every behavioral claim below is grounded in exact source tracing (Read + read-only grep/git), not live-harness runs. Where I would normally have run the fake_s3 harness, I traced the code paths end-to-end instead and flag the one place (s3func's real version-resolution filter) that only a live run can prove. All repos are byte-identical; no files under `~/git/**` were touched.

## 1. Verdict per item

| # | Item | Verdict |
|---|------|---------|
| 1a | exact-key group delete | **Implementable as specified.** Correct fix; one load-bearing implementation detail + a rationale-note correction (below). |
| 1b | two-step `delete_remote`, lock namespace excluded | **Implementable, and an improvement on the adopted roadmap.** One imprecise claim to correct; one efficiency note. |
| 1c | probe relocation into namespace | **Implementable as specified.** Also silently fixes a latent IAM false-negative. |
| 2 | deleted-member recovery → re-check protocol | **Implementable with one required guard** (return `None`, not an empty marker, when no members are actually missing). |
| 3 | `format_version` headroom + document index entry | **Needs change:** exception type should not be `RemoteIntegrityError`; set the attribute in the 404 branch too; the discriminator wording is inaccurate (see F1). |
| 4 | close-time warning on pending `_deletes` | **Implementable as specified.** Minor stacklevel caveat. |
| 5 | declare formats frozen (docs) | **Needs change:** do not freeze the "length==0 ⇒ per-key/metadata" discriminator as written — it is wrong for empty-valued grouped members (F1). |
| 6 | warn on non-HTTPS `db_url` | **Implementable; recommend a single choke point** (S3Connection.__init__ only — a second warning double-fires). |

No item is blocking. The one MAJOR finding (F1) concerns the *documentation/freeze* decision in items 3+5, not the code changes.

## 2. Findings

### F1 — MAJOR — The `length==0` discriminator is not accurate for empty-valued grouped members; do not freeze/document it as "per-key/metadata only"
Items 3 and 5 propose to document and **freeze** the 15-byte index entry with the rule *"length==0 ⇒ per-key/metadata entry, >0 ⇒ grouped"* (the discriminator "already used at utils.py:587"). That rule is only half true.

- A group member's index length is the **value byte-length**: `pack_group` sets `offsets[key] = (pos, len(value))` (`utils.py:77`), and the index entry is `ts(7)+offset(4)+length(4)` (`utils.py:656`). An **empty value** (`eb['k'] = b''`) therefore yields `length==0` for a *grouped* member.
- The push pull-loop treats `length>0` as "has bytes to pull" (`utils.py:587 if length > 0:`). An empty-valued grouped member is skipped from the pull, so if it was evicted locally, `local_file.get_timestamp(key) is None` (`utils.py:613`) → it lands in `lost_keys` → **dropped from the group and `del remote_index[key]`** (`utils.py:617-621`). That is silent data loss for empty-valued grouped keys under evict-then-repack.
- The **read** path actually handles `length==0` correctly (I traced `get_remote_group_values`: the ranged read, verification at `utils.py:373-378`, and `data[rel_offset:rel_offset+0]` all produce `b''`). So the *format* can represent empty values; only the push heuristic and the proposed *documentation* are wrong.

Direction: `length>0 ⇒ grouped` is true, but `length==0 ⇒ per-key/metadata` is false (misses empty grouped members). Before declaring the discriminator frozen (item 5's explicit probe — "is anything NOT good enough to freeze?"), pick one: (a) disallow empty values in grouped mode with a pack-time check, (b) fix the pull heuristic to key off mode+membership rather than `length`, or (c) document the edge honestly and note the eviction-drop as a known bug. Freezing the wording as-is enshrines an inaccurate contract over a latent data-loss path. This is the crisp answer to the item-5 probe. (Pre-existing bug; the round merely risks blessing it.)

### F2 — MINOR — The item-1a rationale note ("failed empty-group delete leaves an orphan invisible to readers") is inaccurate; the fix is actually better than claimed
Tracing the retention logic: on a failed empty-group delete, `upload_group` returns `(error, None)` → `failures[gid]=error` (`utils.py:659`) → `failed_gids={gid}` (`utils.py:666`) → for each deleted key in that group, `continue` **skips** `del remote_index[key]` (`utils.py:668-673`) and retains it. So the pushed index **still references** the (still-existing) group object — a consistent, readable, retryable state, *not* an orphan invisible to readers. The "index no longer references the group" wording describes the **0.9.4 bug** (old `delete_object` swallows errors → `failures` unset → index entry removed → object orphaned) that item 1a *fixes*. Recommend correcting the note so later failure-state reasoning stays accurate. Load-bearing corollary: `delete_object` **must** catch and *return* the error (as specified) — if `delete_objects`' raise (`s3.py:520-525`) were allowed to propagate, `future.result()` (`utils.py:640`) would re-raise and crash the whole push instead of degrading to the retry path. Getting this try/except right is the single most important implementation detail of item 1a.

### F3 — MINOR — `_resolve_missing` empty-marker guard needed for item 2
`recover_group_members` is invoked in three situations (`utils.py:362,386,398`), and the common one (verification/offset mismatch, `utils.py:384-386`) has the object present with **all** members intact — recovery fully succeeds. With `report_missing_members=True`, the function must return `None` when `missing_members` is empty, not `MissingRemoteObject(gid, [])`. An empty-keyed marker is truthy, would trigger a needless index re-pull, and `still_claimed=[]` would route through the clean-absence branch. Guard: `return MissingRemoteObject(...) if missing_members else None`. With that guard, the change is well-targeted: object-present-but-member-genuinely-absent goes loud (the F4 inconsistency from the assessment); object-present-with-stale-offsets stays silent-recover — exactly the intended split. Confirmed there is only one push-path caller to pass `False` (`utils.py:595`); the two read-path callers (`main.py:570`, `main.py:726` via `get_remote_group_value`) keep the default. The bool seam is the minimal correct seam (a marker that always fired would defeat the push self-heal at `utils.py:599-603`). A marker subclass is cleaner long-term but not worth it pre-Phase-1.

### F4 — MINOR — item 3: exception type and 404-branch attribute
- `_load_db_metadata`'s 404 branch (`remote.py:192-197`) sets `_init_bytes/uuid/timestamp/type/num_groups` but would **not** set `format_version` — add `self.format_version = None` (or `1`) there or later code AttributeErrors.
- The "too-new" raise should **not** be `RemoteIntegrityError`: that class means "store contradicts its own index" (`main.py:35-42`), which a version mismatch is not. Use a dedicated `UnsupportedFormatError` (or plain `ValueError`) with a clear "requires a newer ebooklet" message.
- Guard placement in `_load_db_metadata` is correct and complete: it is the single choke point reached by both `S3SessionReader.__init__` (`remote.py:159`), `S3SessionWriter.__init__` (`remote.py:301`), and `_pull_remote_index` (`main.py:613`). No reader/writer is constructed without it. Consequence to accept consciously: a too-new remote also blocks `flag='n'` **replacement** (the guard fires at init, before flag is known) — fine under the no-compat ruling, but worth a one-line acknowledgement.

### F5 — MINOR — item 1b diverges from the adopted roadmap (correctly), with one residual to record
The synthesis roadmap (`architecture-assessment-synthesis.md:35`) specified `delete_remote` as `{exact db_key, db_key+'/', db_key+'.lock.'}` — i.e. it **would** delete the lock namespace. The brief deliberately **excludes** it, with a strong rationale I verified: the lock is created and held for the whole write session (`main.py:214-250`; released only by `ebooklet_finalizer`, `utils.py:114-122`), and `flag='n'` calls `delete_remote()` mid-push under that held lock (`utils.py:529-530`). Today's bare-prefix delete (`remote.py:372`) deletes `D.lock.*` (the lock key is `D.lock.`, `locking.py:166`), destroying the writer's own live tickets and dropping mutual exclusion mid-push. Excluding the namespace is the right call and *improves* on the roadmap. Residual to record: a clean user teardown no longer garbage-collects **crashed-writer** orphan lock tickets (previously swept by the bare prefix), so a recreate-after-crash could inherit stale tickets that block the next writer until `force_lock`. Acceptable (Phase 1 adds age-gated breaking; `force_lock` is the escape hatch), but note it since it's a conscious loss.

### F6 — MINOR — item 1b: "readers see clean absence" is only true for re-resolving readers; and step 1's purge listing is over-broad
- **Ordering is correct** (db-object-first beats children-first): a torn teardown with db-first leaves the db object gone → new readers get `uuid=None` → clean absence; children-first would leave the index claiming now-missing children → `RemoteIntegrityError`. Affirmed.
- **But** a reader holding a *stale* local index that reads a child *after a complete teardown* still goes loud, not absent: child 404 → `MissingRemoteObject` → `_resolve_missing` → `_pull_remote_index` → db object 404 → `fetch_remote_index` returns False (`utils.py:223`) → index **not** swapped (`main.py:631-632`) → old index still claims → `RemoteIntegrityError` (`main.py:687-691`). This is pre-existing (current `delete_remote` also removes the db object) and not worsened by 1b, but the brief's blanket "readers then see clean absence" overstates it. Possible future refinement (defer): when a re-pull finds `uuid` became `None`, treat outstanding markers as whole-remote absence rather than integrity failure.
- **Efficiency:** step 1 `delete_objects(keys=[db_key], purge=True)` resolves the db object's versions via `list_object_versions(prefix=db_key)` (`s3.py:438`), which lists the *entire namespace + siblings* to find one key; step 2 lists again (`prefix=db_key+'/'`). `delete_remote` now does two full listings vs one. Fine for a rare teardown, but a `# over-broad listing to resolve one key` comment (or `purge=False` on step 1) would help.

### F7 — MINOR — item 6: one choke point, not two
Verified the routing: `open_ebooklet`/`open_rcg` → `check_remote_conn` (`main.py:1040`, `1123`), which turns a bare URL string into `S3Connection(db_url=...)` (`remote.py:41`); and `create_s3_read_session` is only ever called from inside `S3Connection` (`remote.py:580`, `remote.py:624`). So **every** path that sets a `db_url` passes through `S3Connection.__init__` — a single `http`-scheme `UserWarning` there covers all public entry points. Adding a *second* warning in `create_s3_read_session` (as the brief hedges) would **double-fire** per session (distinct source locations → the "once per location" filter doesn't dedupe them), because `open()` calls `create_s3_read_session` with the same already-warned `db_url`. Recommend the `__init__`-only choke point. `UserWarning` (non-fatal) is the right channel and is its own escape hatch for local-minio-over-http testing; note the warning is correctly scoped to `db_url` and deliberately not `endpoint_url` (a separate, credential-bearing plaintext concern that's out of scope here — worth one sentence in the release notes).

### F8 — MINOR — item 4: stacklevel cannot be right for both close() paths
`close()` (`main.py:781-786`) is reached directly and via `__exit__` (`main.py:768-769`); the GC weakref finalizer is `ebooklet_finalizer`, **not** `close()` (`main.py:316`), so the warning correctly will *not* fire (unreliably) at GC — and also will not fire at all if the user never calls close/context-exit (accepted; can't warn at GC). A single `stacklevel` will point at user code for one path and at `__exit__` for the other. Minor; pick `stacklevel=2` (correct for direct `close()`, the documented path) and accept the `__exit__` case pointing one frame off. Confirmed `sync()` is local-only (`main.py:789-794`: syncs the two booklets, no remote, no `_deletes` mutation), so the warning after `sync()` sees the true pending count, and a failed-push-then-close correctly still warns (partial push returns early at `main.py:165` without clearing `_deletes`; `update_remote` leaves `retained_deletes` in the set, `utils.py:674-675`). `UserWarning` matches house style (`main.py:293-299`).

### Verification-plan findings
### V1 — MINOR — the fake proves ebooklet's call shape, not s3func's version-filter; keep the live test and widen it
`architecture-assessment-harness-fake_s3.py`'s `delete_objects(keys=...)` just pops exact keys (`:107-110`) and `list_object_versions` aliases `list_objects` with one version per key (`:119-120`). This **does** discriminate the ebooklet-level fix (on 0.9.4, `delete_object` uses `prefix=` → the fake pops `mydb/1,10,12` → test fails; on the fix, `keys=['mydb/1']` → siblings survive → passes; same for `delete_remote` sibling `mydb2` and `mydb.lock.*`). But it does **not** exercise the real safety mechanism — s3func's version-resolution filter `if obj['key'] == key_name` at `s3.py:441` and `:451` — which is what actually makes the exact-keys purge prefix-safe on a real versioned store. Only the one live B2 grouped test covers that. Recommend the live tier also assert (a) `delete_remote` sibling-db survival (`mydb` vs `mydb2`) and (b) lock-ticket survival through `delete_remote`, since the brief's live test only names the emptied-group case; both currently rest on the fake alone.
### V2 — MINOR — FakeLock is a no-op, so lock-survival tests must seed tickets
`FakeLock` (`:34-51`) and `FakeS3Session.lock` (`:132-133`) never write `D.lock.*` objects into the store. The "lock-ticket survival through delete_remote" and "mid-'n'-push own-lock-deletion" regressions therefore require the test to **manually seed** `mydb.lock.<id>-0/1` keys before calling `delete_remote`, then assert they survive. Also note the fake models one version per key, so it can't reproduce all-versions purge, delete markers, or the probe's version-scoped delete — those belong to the live tier.

## 3. What I checked that held

- **Item 1a core fix is correct.** `delete_objects(keys=[db_key+'/'+str(gid)])` with `purge=True` resolves versions via a prefix listing but **filters to the exact key** (`s3.py:438-460`, filters at `:441`/`:451`), so emptying group `1` no longer touches `10/11/12`. I confirmed the bug end-to-end: current `delete_object` uses `delete_objects(prefix=key1)` (`remote.py:352`), and with `num_groups=13` (13 is prime; ids `0..12`) prefix `D/1` matches `D/1,D/10,D/11,D/12`. Under the single-writer invariant there is no concurrent overwrite of the same group key (a group is either uploaded or deleted per push, one future per key), so the eventual-consistency delete-during-overwrite concern does not arise. The only production caller relying on the old prefix behavior is the empty-group delete, which wants *exact* — no legitimate prefix reliance exists (grep confirms callers: `utils.py:307` + one test).
- **Item 1b lock exclusion is correct in both contexts** and the db-first ordering is the right relative choice (see F5/F6). The bounded `D/` prefix in step 2 excludes `D.lock.*` (separator is `.`), sibling `D2` (no `/`), and probe orphans `D<hex>` (no `/`) — verified by prefix arithmetic.
- **Item 1c probe relocation is safe** and a latent fix: nothing lists+parses child keys as group ids (the reader fetches children by exact index-derived key; only `copy_remote` blind-lists at `remote.py:400`, and it merely copies — a transient/failed probe would be a harmless copied orphan). The probe delete is exact key + `version_id` (`remote.py:314`), non-destructive on versioned stores. Relocating into `D/*` also aligns the probe with a namespace-scoped IAM policy (`{D, D/*}`), under which the old `D<hex>` probe would false-negative — a bonus fix. New probe orphans become sweepable by `delete_remote`'s `D/` step; old-location orphans are accepted unsweepable.
- **Item 2 marker plumbing is sound.** I traced that `key in db` currently returns True (`__contains__`, `main.py:465`) while `db[key]` raises `KeyError` (recover materializes present members only, `utils.py:327-331`, `get` returns `_MISSING`, `main.py:738-741`) — the exact inconsistency item 2 removes. The marker flows through `load_items` (`main.py:589-595`) and `_load_item` (`main.py:730-731`) into `_resolve_missing`, whose clean-absence branch is unaffected by the group object still existing (it only re-pulls the index and cleans local state; `s3_key` is used only for logging). Partial materialization (some members recovered, some loud) is safe for `load_items` batching: the pool fully drains before one re-pull, recovered members are already local, the marker carries only the still-missing subset. Writers (holding the lock) correctly stay loud; readers inherit the same accepted mid-push false-positive window already documented at `main.py:669-672` (item 2 extends it to member granularity — a conscious trade of previously-silent-and-wrong for loud-and-mostly-right).
- **Item 3 write side** already emits `type/uuid/timestamp/init_bytes[/num_groups]` at `utils.py:723-730`; adding `format_version:'1'` is a one-line addition. The 15-byte layout (`value_len=15`, `utils.py:240`) and the per-key/metadata `+ b'\x00'*8` construction (`utils.py:643,689`) are as documented — modulo F1.
- **Item 5 RCG schema** at `main.py:890-898` is self-contained and already called a "stable contract"; tightening to "frozen" is safe (the Phase-1 redesign bumps `format_version`, not the RCG entry). No format there is unfit to freeze.

## 4. Adjacent items to absorb or defer

- **Absorb into item 3/5 (small):** resolve F1 before writing the "frozen" wording — at minimum a pack-time guard rejecting empty grouped values, or an honest doc note. This is the one place the round could bless a data-loss path.
- **Absorb into item 1a (trivial):** make explicit in the diff/tests that `delete_object` swallows `urllib3.exceptions.HTTPError` and returns it, never propagates (F2 corollary) — it is the difference between graceful retry and a crashed push.
- **Defer (Phase 1, already roadmapped):** whole-remote-absence handling when a re-pull finds `uuid` gone (F6); crashed-writer lock-orphan sweep (F5) is explicitly the Phase-1 age-gate item.
- **Defer/note:** `endpoint_url` over http is a separate plaintext-credentials exposure out of item 6's scope — one release-note sentence, no code.

Net: the six changes are well-designed and mostly implementable as specified. The only substantive push-back is F1 (don't freeze an inaccurate discriminator over a real empty-value drop), plus small corrections to the item-1a rationale note (F2), the item-2 empty-marker guard (F3), the item-3 exception type/attribute (F4), and the item-6 single-choke-point (F7). Items 1b and 1c are, if anything, better than the adopted roadmap.
